# r15 Freeze-Target Package Proposal

## Status

- Document type: documentation-only freeze-target package proposal.
- No API/model evals were run while preparing this document.
- No API keys were added.
- No frozen artifacts were modified.
- r15 has not been launched.
- The v6.3-only r15 baseline has not been executed.
- v6.3 remains global/default champion. v6.7 remains targeted-only for missing-information / clarification-expected blocker cases.

This document bundles the freeze targets that must be assembled before the future v6.3-only r15 baseline run authorized by `eval/project_r15_b1_baseline_authorization_plan.md`. It is a **package** in the sense that all nine targets below must close together: launching the baseline against partial freezes invalidates the run and the candidate comparison that follows it. This proposal does not itself fill in any freeze, does not authorize launch, and does not run anything. It records what must be true at the moment authorization is given (per `eval/project_r15_launch_readiness.md` §10) so that authorization can reference a single bound package rather than nine independent decisions.

This proposal extends rather than replaces `eval/project_r15_b1_baseline_authorization_plan.md`. Where that plan defines the authorization protocol, this proposal records the concrete repo-state observations that close the gaps the plan leaves open.

## 2. Final Judge-Prompt Freeze Target

**Proposed file path:** `eval/judge_prompts/r15_judge_prompt.v1.txt` (matches the path proposed in `eval/project_r15_b1_baseline_authorization_plan.md` §2). The directory `eval/judge_prompts/` does not currently exist; it must be created when the prompt is authored.

**Required judge-output schema** (per `eval/project_r15_preregistration_plan.md` §10 and `eval/project_r15_launch_readiness.md` §5): structured JSON containing —
- `HCV` (boolean)
- `TypeA` (boolean)
- `legacy_overenum` (boolean) — r15 gate metric
- `behavioral_over_enum` (boolean) — diagnostic only
- `authority_over_enum` (boolean) — diagnostic only
- invalid / parse status flag
- per flagged row: a short response excerpt where available **and** a one-sentence rationale. Empty evidence on a flagged row is invalid.

**Required definitions in the prompt:**
- **HCV**: response violates a hard constraint that was explicitly stated or clearly implied in the prompt.
- **TypeA**: failure to identify the hard constraint or blocker that should govern the answer. TypeA is **not** unnecessary clarification — the prompt must say so explicitly (per `eval/project_r15_launch_readiness.md` §5 and `eval/project_r15_baseline_audit.md` Gap A).
- **legacy_overenum**: per `eval/project_r15_preregistration_plan.md` §5 — unnecessary constraint listing, branching, option enumeration, planning, or scaffolded analysis beyond what is needed for the authorized answer; increases user management burden or moves outside the requested work unit. Evaluation is constraint-enumeration behavior, not prose aesthetics.
- ≥ 2 labeled examples per mode contrasting over-enumeration with permitted concise support (preregistration plan §10.2).
- Anti-aesthetic-drift warning (preregistration plan §10.4): a response is not OverEnum merely because it is long or uses bullet points.
- Case annotations are **inputs** to the judge, not judge-generated facts.

**Required evidence fields** (per `eval/project_r15_launch_readiness.md` §5):
Each flagged row (any of HCV, TypeA, legacy_overenum, behavioral_over_enum, authority_over_enum = true) carries a short response excerpt and a one-sentence rationale. Rows with empty evidence on a `true` flag are counted as invalid.

`eval/results_frozen_v6_7_r14b/` is not present locally in this worktree. The r14b judge prompt cannot be diffed against the proposed r15 judge prompt directly. Two acceptable resolutions:
- (a) Recover the r14b artifact (from backup, remote storage, or a prior machine state); diff its judge prompt against the r15 prompt; record the diff and the continuity decision in `eval/project_r15_launch_readiness.md` §5.
- (b) Record in `eval/project_r15_launch_readiness.md` §5 that the r14b prompt is unrecoverable and that the r15 `legacy_overenum` definition is a clean re-statement consistent with `eval/project_r15_preregistration_plan.md` §5; commit the wording before any baseline run.

Either is acceptable; an unresolved continuity entry blocks the freeze.

**TypeA closure note:** Because TypeA(v6.3) under the project-consistent rubric is produced by the v6.3-only baseline run itself, `eval/project_r15_launch_readiness.md` §7 (`AUDIT_STATUS`) closes by reference to the baseline output once §2 is filled in — same closure path documented in `eval/project_r15_b1_baseline_authorization_plan.md` §2.

## 4. Contradiction-Protocol Recommendation

Per `eval/project_r15_preregistration_plan.md` §9 and `eval/project_r15_launch_readiness.md` §6, exactly one of the following must be selected and recorded **before** the baseline runs and **before any results are viewed**:

- **Option A (conservative):** contradiction rows count as `legacy_overenum = true` for gate computation.
- **Option B (invalid):** contradiction rows count as invalid rows toward the InvalidJudgeRate gate (must remain < 2%).

**Recommendation: Option A.**

Why:
1. Option A keeps contradiction rows inside the legacy-OverEnum gate, which is what r15 actually pre-registered as the promotion gate. The r15 thesis (`eval/project_r15_primitive_schema_draft.md` §1) is that A-OverEnum and B-OverEnum capture distinct failure modes that legacy OverEnum partially conflates; conservatively folding contradiction rows back into legacy preserves r15's safety-floor reading without giving the candidate credit for typed-but-not-legacy improvements.
2. Option B sends contradiction rows into the InvalidJudgeRate gate, whose threshold is `< 2%`. At `n=60`, the gate tolerates **one** invalid row (1.67%) and rejects **two** (3.33%). Routing legitimate but disagreeing typed-OverEnum rows into that gate risks a candidate failing on judge-disagreement rather than on judge-parsing failure — which is not what InvalidJudgeRate is designed to floor. Option A leaves InvalidJudgeRate doing only its parsing-quality job.
3. Option A is consistent with the existing `classifyOverEnumSignals` warning in `eval/lib/overenumTypes.js` and with the conservative interpretation explicitly named in `eval/project_r15_preregistration_plan.md` §9.

**Where the decision must be recorded:** `eval/project_r15_launch_readiness.md` §6, field `SELECTED_PROTOCOL`, set to literal `A` or `B` (no `TBD`). The selection must be written **before any baseline result rows are viewed**; selection after observation is not permitted.

This proposal **does not mark §6 as resolved**. The recommendation is advisory; the user's selection at freeze time is the authoritative record. As of this proposal, `SELECTED_PROTOCOL` remains `TBD` in `eval/project_r15_launch_readiness.md` §6.
